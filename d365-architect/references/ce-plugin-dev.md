## Event Pipeline — Four Stages (Official)

| Stage | Code | In Transaction | Use For | Never Use For |
|---|---|---|---|---|
| PreValidation | 10 | Before tx opens* | Cancel operation; runs before security checks | Reading related records not yet in context |
| PreOperation | 20 | Yes (in tx) | Modify Target values before save; set defaults | Cancelling (causes rollback + perf hit); external calls |
| MainOperation | 30 | Internal only | Custom API / virtual table providers only | Direct plugin registration |
| PostOperation | 40 | Yes (sync) / No (async) | Modify response properties; trigger side effects | Updating same entity (triggers new Update event) |

> **Doc note:** "An exception thrown at any synchronous stage within the database
> transaction causes the entire transaction to roll back. Cancel operations in
> PreValidation, not PreOperation." (Event Framework docs)

> **Async PostOperation:** Register as `ExecutionMode=Asynchronous` to run
> outside the database transaction via the async service. Required for
> `SystemUser` Create event updates. (Event Framework docs)

## IPlugin Interface — Official Pattern

Plug-in must target **.NET Framework 4.6.2** (official spec). Target
`netstandard2.0` for cross-platform build tooling; it is compatible.

```csharp
// Official IPlugin pattern — from MS Docs "Write a plug-in"
// RULE: Class must be stateless — no instance fields that store service
// references or context data (BP: develop IPlugin implementations as stateless)
public class OnCreateOrder : IPlugin
{
    // Constructor overloads for configuration injection (optional)
    public OnCreateOrder() { }
    public OnCreateOrder(string unsecure) { }            // visible config
    public OnCreateOrder(string unsecure, string secure) { } // secure config

    public void Execute(IServiceProvider serviceProvider)
    {
        // 1. Execution context — stage, message, entity, depth, initiating user
        var context = (IPluginExecutionContext)serviceProvider
            .GetService(typeof(IPluginExecutionContext));

        // 2. Tracing — ALWAYS get this first; required for diagnostics
        //    BP: "Use ITracingService in Plug-ins"
        var tracingService = (ITracingService)serviceProvider
            .GetService(typeof(ITracingService));

        // 3. Org service factory → org service
        //    NEVER use Web API inside plugins (not supported per docs)
        var factory = (IOrganizationServiceFactory)serviceProvider
            .GetService(typeof(IOrganizationServiceFactory));
        var orgService = factory.CreateOrganizationService(context.UserId);

        // Guard: verify message and entity (defensive; also prevents
        // firing on re-use of registered step for wrong entity)
        if (context.MessageName != "Create" ||
            context.PrimaryEntityName != "dexx_order")
            return;

        var target = (Entity)context.InputParameters["Target"];

        try
        {
            tracingService.Trace("OnCreateOrder: start, Id={0}", target.Id);
            // Business logic here
        }
        catch (FaultException<OrganizationServiceFault> ex)
        {
            // BP: "Use InvalidPluginExecutionException in plug-ins"
            throw new InvalidPluginExecutionException(
                "OnCreateOrder failed: " + ex.Detail.Message, ex);
        }
        catch (Exception ex)
        {
            tracingService.Trace("OnCreateOrder: unhandled error: {0}", ex);
            throw new InvalidPluginExecutionException(
                "OnCreateOrder unexpected error: " + ex.Message, ex);
        }
    }
}
```

## PluginBase Pattern (Modern — pac plugin init)

Official alternative: derive from `PluginBase` generated by `pac plugin init`
or Power Platform Tools for VS. Recommended for new projects.

```bash
# Generate PluginBase project via pac CLI (official tooling)
pac plugin init --skip-signing
```

```csharp
// PluginBase pattern — cleaner service access via LocalPluginContext
public class OnUpdateContact : PluginBase
{
    public OnUpdateContact(string unsecureConfig, string secureConfig)
        : base(typeof(OnUpdateContact)) { }

    protected override void ExecuteDataversePlugin(
        ILocalPluginContext localPluginContext)
    {
        if (localPluginContext == null)
            throw new ArgumentNullException(nameof(localPluginContext));

        var context        = localPluginContext.PluginExecutionContext;
        var orgService     = localPluginContext.ServiceClient;
        var tracingService = localPluginContext.TracingService;

        // BP: "Include filtering attributes with plug-in registration"
        // Set FilteringAttributes in PRT to avoid firing on every Update
        tracingService.Trace("OnUpdateContact: depth={0}", context.Depth);

        try
        {
            var target = (Entity)context.InputParameters["Target"];
            // Business logic
        }
        catch (FaultException<OrganizationServiceFault> ex)
        {
            throw new InvalidPluginExecutionException(ex.Detail.Message, ex);
        }
    }
}
```

## Key Context Properties

```csharp
context.MessageName          // "Create", "Update", "Delete", "Associate", etc.
context.PrimaryEntityName    // logical entity name, e.g. "account"
context.PrimaryEntityId      // GUID of target record
context.Stage                // 10=PreVal, 20=PreOp, 40=PostOp
context.Mode                 // 0=Sync, 1=Async
context.Depth                // call depth — guard against infinite loops
context.InitiatingUserId     // real user who triggered; use for dual-write guard
context.UserId               // executing user (may differ with impersonation)
context.InputParameters      // "Target" (Entity), "EntityMoniker" for Delete
context.OutputParameters     // "BusinessEntity" for Retrieve output
context.PreEntityImages      // snapshot before operation (register in PRT)
context.PostEntityImages     // snapshot after operation (register in PRT)
context.ParentContext         // parent pipeline context if triggered by another plugin
context.SharedVariables      // pass data downstream to other plugins same pipeline
```

> **Depth guard** — prevent infinite recursion when your plugin calls Org Service
> and triggers itself: `if (context.Depth > 1) return;`

## Critical Best Practices (from official BP docs)

```
BP-1  Stateless: no instance fields storing services or context data
BP-2  InvalidPluginExecutionException for ALL user-facing errors
BP-3  ITracingService: trace at entry, exit, and all branch decisions
BP-4  No ExecuteMultiple/ExecuteTransaction inside plugins
BP-5  No parallel/multi-thread execution inside plugins
BP-6  No duplicate step registration (causes multiple firings)
BP-7  Set FilteringAttributes on Update steps (avoid fire on every Update)
BP-8  Set Timeout on external calls — external HTTP has no default timeout
BP-9  Set KeepAlive=false on HttpWebRequest to external hosts
BP-10 Single assembly per solution (manage plug-ins in single solution)
BP-11 Avoid Retrieve/RetrieveMultiple synchronous steps for perf
BP-12 Merge separate plugins into one assembly for perf/maintainability
```

## Scaffold + Build Commands

```bash
# Option A: pac plugin init (official, generates PluginBase)
pac plugin init --skip-signing
dotnet build

# Option B: manual classlib (cross-platform)
mkdir -p src/plugins/MyPlugin && cd src/plugins/MyPlugin
dotnet new classlib -n MyPlugin --framework netstandard2.0
dotnet add package Microsoft.CrmSdk.CoreAssemblies --version 9.0.2.62
dotnet add package Microsoft.CrmSdk.Workflow --version 9.0.2.62
dotnet build --configuration Release

# Strong-name sign (required for Sandbox isolation)
sn -k MyPlugin.snk
# Add to .csproj:
# <AssemblyOriginatorKeyFile>MyPlugin.snk</AssemblyOriginatorKeyFile>
# <SignAssembly>true</SignAssembly>

# Register with Plug-in Registration Tool (PRT) via pac
pac tool prt
```

## On-Premises Specific Notes (v9.1)

- On-disk plugins (full-trust): place assemblies in `<installdir>\Server\bin\assembly`
- **Don't depend on variable references passed into plugins** — data is
  serialized/deserialized in the pipeline; object instances are new copies.
  Changes to a QueryExpression inside a plugin don't reflect on the original.
  (Source: on-premises plugin development doc)
- Offline plugin execution supported for on-premises deployments
- Full-trust plugins run within the same app domain on-premises
